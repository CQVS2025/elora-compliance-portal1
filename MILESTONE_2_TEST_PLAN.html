Elora Marketplace — Milestone 2 End-to-End Test Plan

Complete test plan for Milestone 2 (checkout, payments, fulfilment, Xero, email). Covers everything from a clean environment setup through to live production verification. Follow it top-to-bottom on a fresh Supabase project (staging recommended); each section ends with explicit pass criteria.

Configuration discipline: every Edge Function reads its credentials from Supabase project secrets via Deno.env.get(...). Do NOT use a local .env file for any M2 secret.

2 · One-Time Environment Setup

2.1 Set all Supabase project secrets

These must be set via Supabase project secrets, NOT in a .env file.

supabase login
supabase link --project-ref <project-ref>

supabase secrets set \
  MAILGUN_API_KEY=key-************************ \
  MAILGUN_DOMAIN=eloracomplienceprotal.com \
  MAILGUN_BASE_URL=https://api.mailgun.net \
  MARKETPLACE_EMAIL_FROM='Elora Marketplace <orders@eloracomplienceprotal.com>' \
  MARKETPLACE_PORTAL_URL=https://www.eloracomplienceprotal.com \
  XERO_CLIENT_ID=********************************* \
  XERO_CLIENT_SECRET=********************************* \
  XERO_REDIRECT_URI=https://<project-ref>.functions.supabase.co/marketplace_xero_oauth_callback \
  STRIPE_SECRET_KEY=sk_test_*********************** \
  STRIPE_WEBHOOK_SECRET=whsec_********************* \
  GOOGLE_DISTANCE_MATRIX_API_KEY=AIzaSy*********************
MAILGUN_BASE_URL is optional (defaults to https://api.mailgun.net); set it to https://api.eu.mailgun.net if the Mailgun account is on the EU region. MARKETPLACE_EMAIL_FROM is optional; if unset, the sender falls back to Elora Marketplace <postmaster@${MAILGUN_DOMAIN}>. GOOGLE_DISTANCE_MATRIX_API_KEY is optional but strongly recommended: when set, freight quotes use real Google road km; when unset, the engine silently falls back to haversine × 1.25 (which under-charges on hilly / island routes).

Verify with supabase secrets list — all 8 required keys must appear (plus any optional ones you set).

2.2 Pass criteria

  • supabase secrets list shows all 8 required keys (MAILGUN_API_KEY, MAILGUN_DOMAIN, MARKETPLACE_PORTAL_URL, XERO_CLIENT_ID, XERO_CLIENT_SECRET, XERO_REDIRECT_URI, STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET)
  • GOOGLE_DISTANCE_MATRIX_API_KEY is set (or you have explicitly accepted that freight quotes will use the haversine × 1.25 fallback)
  • No .env file contains M2 secrets in the deployed environment

8 · Functional Test Flows

8.1 Flow A — Stripe (card) checkout happy path (deferred order creation)

Actorsbuyer1.user@acme.test, mpadmin@elora.test, warehouse@elora.test

Critical contract change: the order row in marketplace_orders is created ONLY after Stripe confirms the payment (via webhook). Before payment, only a marketplace_checkout_sessions row exists with the calculated intent_payload. The buyer's cart is NOT wiped until success. This mirrors Chem Connect's /api/orders/finalize pattern.
  1. Buyer signs in, adds 2 products to cart (one with a pricing override).
  2. Go to /marketplace/checkout.
  3. Pick a delivery postcode that falls inside the first freight bracket.
  4. Answer site-access questions.
  5. Choose Pay by card (Stripe) and click Checkout.
  6. Use Stripe test card 4242 4242 4242 4242, any future expiry, any CVC.
  7. On success, browser redirects to /marketplace/checkout/success?session=<checkout_session_id>. The page polls the session row every 2s for up to 30s while the webhook materialises the order, then renders View order.

Verify in DB (after webhook fires):

select id, order_number, status, payment_method, stripe_session_id, stripe_paid_at,

from marketplace_orders order by created_at desc limit 1;

select id, order_id, status, stripe_payment_intent_id
from marketplace_checkout_sessions order by created_at desc limit 1;

  • checkout_sessions: status='completed', order_id now NON-NULL (was NULL before webhook), stripe_payment_intent_id set
  • cart: marketplace_cart_items for the buyer is empty (webhook wiped it after materialising)
